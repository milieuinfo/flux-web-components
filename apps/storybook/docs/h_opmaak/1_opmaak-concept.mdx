## Concept

Sinds de v2-release zijn de `element`-componenten weggewerkt. Dit heeft impact op de globale styling.

- In **v1**:  
  Componenten zoals `vl-button` moesten gebruikt worden als:  
  ```html
  <button is="vl-button">Actie</button>
  ```
  Deze hadden geen shadow DOM. CSS werd dus globaal toegepast.

- In **v2**:
  Componenten zijn nu volwaardige web-componenten, bv.:

  ```html
  <vl-button>Actie</vl-button>
  ```
  Deze hebben een shadow DOM. De styling van het component zelf zit in die shadow DOM. Externe CSS “lekt” daar niet 
  automatisch in. Om tweaks mogelijk te maken voorzien de meeste componenten een `custom-css`-attribuut.

## Base (intern)

Dit is basis CSS, bouwstenen die **intern** gebruikt worden.
Afnemers gebruiken dit nooit rechtstreeks.
Deze CSS wordt ofwel globaal op het document gezet of in web-componenten toegepast.

Inhoud van de base-styles:

- [Vlaanderen fonts](/docs/styles-base-font--documentatie)
- [Mixins](/docs/styles-base-mixin--documentatie) (bv. animaties)
- [Variabelen](/docs/styles-base-var--documentatie) (kleuren, media-queries, …)
- Styling voor native elementen, zoals [body](/docs/styles-base-body--documentatie)

## Layout (afnemers)

Dit is de basis voor algemene layout: alles wat orde en structuur geeft. Het zijn **containers** waarin andere 
componenten geplaatst worden.

In **v1** waren dit `element`-implementaties. Sinds **v2** zijn dit **CSS-classes** op native elementen.

Waarom geen web-componenten?

- CSS en HTML vormen de basis van de layout en opmaak van een web-pagina. Het is aan te raden deze zoveel mogelijk 
  rechtstreeks toe te passen zonder JavaScript.
- Containers zijn niet visueel, maar bepalen enkel plaats en ruimte.
- Met CSS op native elementen kan er zoveel genest worden als nodig, zonder shadow DOM problemen.

### Beschikbare classes

- `.vl-content-block` – om content te groeperen
- `.vl-grid`, `.vl-column` – 12-koloms layout
- `.vl-group` – groepeert componenten horizontaal of verticaal
- `.vl-margin-...`, `.vl-padding-...`, `.vl-spacer-...` en `.vl-stacked-...` – witruimte aanpassen
- `.vl-section` – structuur definiëren
- `.vl-separator-...` – varianten op de horizontale regel

### Voorbeeld

```html
<section class="vl-section">
  <div class="vl-content-block">
    <div class="vl-grid">
      <div class="vl-column vl-column--6">
        <!-- Slechts op dit niveau is er nood aan een web-component -->
        <vl-info-tile>...</vl-info-tile>
      </div>
    </div>
  </div>
</section>
```

### Afname

Afnemers hoeven niets specifiek te doen om deze classes in te laden.
Zodra er één Flux-component gebruikt wordt, wordt alle layout-CSS globaal geregistreerd.

Indien je in je toepassing ook zelf web-componenten bouwt en je binnen je component de Flux layout styles wilt 
gebruiken, moet je de `vlLayoutStyles` manueel registreren adhv de `styles()`-method:

```ts
import { vlLayoutStyles } from '@domg-wc/styles';

// Binnen de component class:
static get styles() {
  return [
    vlLayoutStyles,
  ]
}
```

## Aanbevolen volgorde van styling

Bij het toepassen van styling raden we deze volgorde aan:

1. **Gebruik de Flux-styles**

  - Layout-classes zoals `.vl-padding-...`, `.vl-margin-...`, `.vl-group`, `.vl-section`, …
  - In de meeste gevallen voldoende voor een nette opmaak.

2. **Tweaken met custom-css**

  - Voor kleine afwijkingen of correcties.
  - Hou dit beperkt en consistent.
  - Dit wordt door de meeste componenten ondersteund via het `custom-css` attribuut. Deze CSS wordt toegevoegd aan de 
    `adoptedStyleSheets` in de shadow DOM van de component.
  - Voorbeeld
    ```html
    <vl-info-tile custom-css="h3 { color: yellow; }">...</vl-info-tile>
    ```

3. **Schrijf custom style classes**

  - Voor het stylen / layouten van elementen en pagina's die specifiek gebruikt worden in de toepassing (voor grote 
    of herbruikbare aanpassingen).
  - Voor het aanpassen van Flux CSS-variabelen:
    - CSS-variabelen kunnen globaal aangepast worden en worden ook toegepast binnen de shadow DOM van web-componenten.
    - CSS-variabelen kunnen in een custom class aangepast worden om enkel een deel van de pagina te beïnvloeden.
  - Gebruik een duidelijke prefix (bv. `app-`) om conflicten te vermijden.
  - Injecteer deze classes in de `adoptedStyleSheets` van je toepassing.
  - Voorbeeld
    ```ts
    export class ToepassingMain extends BaseLitElement {
      static get styles() {
        return [
          vlLayoutStyles,
          css`
            .toepassing-main { 
              --vl-side-sheet-width: 600px;
            }
          `,
        ]
      }

      render() {
        return html`
          <main class="toepassing-main">...</main>
        `
      }
    }
    ```

## Utility classes (Digitaal Vlaanderen)

De [layout utitily classes](https://overheid.vlaanderen.be/webuniversum/v3/documentation/utilities/) van Digitaal 
Vlaanderen worden niet as-is overgenomen en niet standaard beschikbaar gemaakt in de Flux web-componenten. Sommige van 
deze utilities hebben we overgenomen als web-component indien daar een use-case voor was. Gebruikers kunnen ontbrekende 
styles overnemen in custom classes (zoals [hierboven](#aanbevolen-volgorde-van-styling) uitgelegd wordt) en ze op die 
manier toepassen in hun applicatie. 

## Conventies

- Alle CSS wordt geschreven als **CSS-in-TS** en aangeboden als **CSS-in-JS**.
- De CSS volgt de [**BEM**](https://getbem.com/)-methodologie.
- CSS-variabelen worden gebruikt om styling eenduidig te specificeren.
  - Variabelen zijn globaal gedefinieerd.
  - Lokale afwijkingen zijn mogelijk binnen een component.
- Mixins worden geïmplementeerd als TypeScript-functies die parametriseerbare CSS teruggeven.
